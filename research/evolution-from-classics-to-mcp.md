# Multi-Agent Coordination: Protocol Stack 2026

> From KQML (1993) to the Agentic AI Foundation (2025): how the industry converged on a layered, complementary protocol stack for agent coordination.

---

## The 2026 Protocol Stack

The industry has converged on a **four-layer** architecture. No single protocol covers everything.

```
Layer 4: DISCOVERY (Open Internet)     --> ANP (W3C DID-based, early stage)
Layer 3: AGENT-TO-AGENT                --> A2A (Google, 150+ orgs, LF AI & Data)
Layer 2: AGENT-TO-TOOL                 --> MCP (Anthropic, 10K+ servers, AAIF)
Layer 1: INTRA-APP ORCHESTRATION       --> SDKs (LangGraph, CrewAI, OpenAI, etc.)
```

MCP and A2A are **complementary**, not competing. MCP handles how agents access tools and data; A2A handles how agents talk to each other. Most production systems use both.

---

## Layer 2: MCP (Model Context Protocol)

**Origin**: Anthropic, November 2024. Donated to the Agentic AI Foundation (Linux Foundation) December 2025.
**Status**: Industry standard. 10,000+ public servers. Adopted by ChatGPT, Cursor, Gemini, VS Code, Claude.
**Spec Version**: 2025-11-25

### What It Does

Standardizes how an AI agent connects to external **tools, data sources, and services** (vertical integration: agent-to-world).

### Core Primitives

**Server-provided**:
| Primitive | Direction | Description |
|-----------|-----------|-------------|
| Tools | LLM-controlled | Functions the agent can invoke (like POST endpoints) |
| Resources | App-controlled | Data/content for LLM context (like GET endpoints) |
| Prompts | User-controlled | Reusable prompt templates |

**Client-provided**:
| Primitive | Direction | Description |
|-----------|-----------|-------------|
| Sampling | Server-initiated | Server requests LLM completions back (reverse invocation) |
| Elicitation | Server-initiated | Server asks users for input |
| Roots | Server-initiated | File system boundaries the client grants |

**Transport**: JSON-RPC 2.0 over stdio (local) or HTTP+SSE (remote).

### November 2025 Updates
- Async operations via experimental Tasks primitive (states: working, input_required, completed, failed, cancelled)
- Statelessness option alongside stateful sessions
- Server identity verification
- Official community-driven registry for MCP server discovery

## Layer 3: A2A (Agent-to-Agent Protocol)

**Origin**: Google, April 2025. IBM's ACP merged into A2A August 2025. Hosted under LF AI & Data.
**Status**: Production-ready for early adopters. 150+ supporting organizations. v0.3 (July 2025).

### What It Does

Standardizes how **agents communicate with each other** (horizontal integration: agent-to-agent). Agents are opaque -- no access to internal state, memory, or tools.

### Core Concepts

| Concept | Description |
|---------|-------------|
| **Agent Card** | JSON at `/.well-known/agent.json`. Declares identity, capabilities, skills, endpoint, auth requirements. |
| **Task** | Central unit of work. Lifecycle: submitted -> working -> input_required -> completed / failed / cancelled |
| **Message** | Communication between client and remote agent. Contains role + one or more Parts. |
| **Part** | Content unit: TextPart, FilePart (inline or URI), DataPart (structured JSON) |
| **Artifact** | Output generated by the remote agent |

### Operations

| Operation | Description |
|-----------|-------------|
| SendMessage | Initiate task with remote agent |
| SendStreamingMessage | Same with SSE/gRPC streaming |
| GetTask / ListTasks | Retrieve task state |
| CancelTask | Request termination |
| SubscribeToTask | Persistent progress stream |
| Push Notifications | Webhook-based async delivery |

### Discovery

Agent Cards may be digitally signed (JWS with JCS canonicalization). Extended Agent Cards return different details based on client auth level. Discovery via: well-known URI, curated registries, or direct configuration.

### Transport (v0.3)

Three options (all must provide identical functionality): JSON-RPC 2.0, gRPC, REST.

## Layer 4: ANP (Agent Network Protocol)

**Origin**: Open-source community (China Mobile, Huawei backing). IETF Internet-Draft submitted 2025.
**Status**: Early stage / research. Most relevant for open-internet, decentralized scenarios.

### Architecture

| Layer | Function |
|-------|----------|
| Identity & Encrypted Communication | W3C DID-based (`did:wba`). Trustless authentication, E2E encryption. |
| Meta-Protocol Negotiation | Dynamic negotiation of communication protocols |
| Application Protocol | Task-level interaction using negotiated protocols |

Discovery via crawled Agent Description Protocol (ADP) endpoints and DID resolution.

---

## Discovery Standard: AGENTS.md

**Origin**: OpenAI, August 2025. Now governed by AAIF.
**Status**: 60,000+ open-source projects. Supported by GitHub Copilot, Cursor, Gemini CLI.

Simple markdown files guiding AI coding agents -- a "README for agents." Not a wire protocol, but a discovery and instruction standard. Analogous to Kapi's own CLAUDE.md.

## Classical Roots

The modern protocols didn't emerge from nothing. They're reinterpretations of 30 years of multi-agent systems research:

### KQML (1993)
- DARPA Knowledge Sharing Effort
- Performatives (speech acts): ask-one, tell, achieve, advertise, subscribe, broker
- Three-layer messages: content, message, communication
- Limitation: loose semantics -- agents could interpret performatives differently

### FIPA ACL (2000)
- Refined KQML with formal BDI semantics (Belief-Desire-Intention)
- Communicative acts: inform, request, agree, refuse, propose, cfp, accept-proposal, reject-proposal
- Standardized interaction protocols (reusable conversation patterns)
- Limitation: formal complexity was impractical; the web era favored lighter approaches

### Contract Net Protocol (Smith, 1980)
- Manager broadcasts task -> Contractors bid -> Manager awards contract
- The foundational task allocation protocol
- Modern equivalent: supervisor-worker delegation in LangGraph, ADK coordinator

### Interaction Protocol Patterns (FIPA)

| Protocol | Pattern | Modern Equivalent |
|----------|---------|-------------------|
| Request | A requests -> B agrees/refuses -> returns result | Simple tool call / handoff |
| Contract Net | Announce -> Bid -> Award | Router dispatching to specialists |
| Brokering | A asks broker -> Broker finds provider | A2A Agent Card discovery |
| Subscribe | A subscribes -> B sends updates | MCP resource subscriptions |
| Propose | A proposes -> B accepts/rejects | Generator/Critic pattern |

### Blackboard Systems (Erman, 1980)
- Shared knowledge space where specialists read/write
- Controller decides which agent acts based on blackboard state
- Modern equivalent: LangGraph shared state, Redis-backed coordination

### Shared Plans / Joint Intentions
- SharedPlans (Grosz & Kraus): agents form plans with mutual beliefs about goals
- Joint Intention Theory (Cohen & Levesque): commit to goals together, obligation to inform teammates of changes
- Key insight: coordination requires **mutual belief** -- knowing the plan AND knowing that others know it

### What Changed from Classical to Modern

| Classical | Modern | What's Better | What's Worse |
|-----------|--------|--------------|-------------|
| Formal BDI semantics | Natural language + JSON schemas | Flexibility, no shared ontology needed | Loss of formal guarantees |
| FIPA performatives | Typed message parts (Text/File/Data) | Simpler, REST-native | Less semantic precision |
| Hand-crafted knowledge bases | LLM world knowledge | Adaptability, generalization | Hallucination, inconsistency |
| Shared ontologies required | LLMs handle semantic alignment | Lower engineering overhead | Occasional ambiguity |
| Polynomial communication scaling | Same fundamental problem | - | Worse -- LLM inference is expensive |

The emerging synthesis: use LLMs for flexible communication + reintroduce structured coordination protocols (JSON schemas, typed messages) for reliability-critical interactions.
